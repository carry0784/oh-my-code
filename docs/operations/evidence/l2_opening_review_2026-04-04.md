## 2. 장점 / 단점

### L2 개방 시 장점

- 읽기 전용 관측 API 추가/수정이 가능해져 **가시성 확장** 가능
- 대시보드 조회 기능, 로그 출력 추가 등 **운영 관찰력 향상**
- Gate는 여전히 LOCKED이므로 L3/L4는 차단 유지 — **정책 변경 위험 없음**
- 4444/4444 PASS + drift 0 + 금지 침범 0이므로 **근거 충분**

### L2 개방 시 단점/리스크

- L2 범위 내에서도 부주의한 변경이 기준선 관측을 흐릴 수 있음
- 단, L2는 "읽기 전용 / 관측 전용"이므로 write path 리스크는 없음

### L2 미개방 유지 시

- 추가 안정 관찰 가능하나, 이미 충분한 검증 완료
- 불필요한 지연 외 추가 이점 없음

## 3. 이유 / 근거

### 근거 1 — 24시간 체크리스트 4개 조건 전부 충족

| 조건 | 검증 결과 |
|------|-----------|
| 신규 실패 0 | 4444/4444 PASS, 이전 대비 신규 실패 없음 |
| drift 0 | 6항목 전부 일치 (GUARDED_RELEASE, DATA_ONLY, 5, 3, absent, consistent) |
| Gate 위반 0 | L3/L4 무단 변경 없음 (L2 선반영 건은 EX-001로 별도 종결) |
| 금지 범위 침범 0 | 금지 task 미발송, private API 미호출 |

### 근거 2 — L2 범위의 정의가 명확

Change Gate Policy에 따른 L2 허용/금지:

| L2 허용 | 조건 |
|---------|------|
| 읽기 전용 API 추가 | 기준선 6항목 불변 + baseline-check PASS |
| 로그 출력 추가 | 상태 변경 없음 |
| 대시보드 조회 기능 추가 | 쓰기 경로 없음 |

| L2 금지 | 사유 |
|---------|------|
| API에서 ops_state.json 쓰기 경로 추가 | A 전용 편집권 침해 |
| /status 응답 구조 변경 (기존 필드 제거) | 관측 경로 훼손 |

L2를 열어도 **write path, 실행 경로, 정책 변경은 전부 차단 유지**됩니다.

### 근거 3 — 보호 장치가 충분

L2 개방 후에도 유지되는 통제선:

| 통제 | 상태 |
|------|------|
| Gate LOCKED | L3/L4 차단 유지 |
| DATA_ONLY 계약 | 유지 |
| 금지 task 3개 | 미발송 유지 |
| blocked API 5개 | 차단 유지 |
| CR-046 관측 | 별도 유지 |
| CR-049 Phase 3 | DESIGN_ONLY 유지 |
| GR-RULE-01~03 | 적용 중 |
| 재진입 조건 | drift/회귀 실패 시 즉시 Hold 재진입 |

## 4. 실현 / 구현 대책

### L2 개방 시 반영 사항

| 항목 | 현재 → 변경 |
|------|------------|
| `ops_state.json` `allowed_scope` | `["L0", "L1"]` → `["L0", "L1", "L2"]` |
| `ops_state.json` `pending_scope` | `["L2"]` → `[]` |
| `/change-gate` 응답 `reentry_allowed["L2"]` | `false` → `true` |
| Gate 상태 | **LOCKED 유지** (L3/L4 계속 차단) |

### L2 개방 후에도 계속 차단

| 항목 | 상태 |
|------|------|
| L3 (런타임) | **차단** — A 승인 필요 |
| L4 (정책/실행) | **차단** — 별도 CR 필요 |
| Gate | **LOCKED** |
| 금지 사항 전체 | **유지** |

### `/change-gate` 코드 변경 (L2 개방 반영)

`ops.py`에서 `reentry_allowed["L2"]`를 `GUARDED_RELEASE`에서도 `True`로 변경해야 합니다. 이 변경 자체도 L2이지만, L2 개방 승인 후에 수행하면 범위 내입니다.

## 5. 실행방법

### A에게 요청하는 판정

| 판정 | 내용 |
|------|------|
| **APPROVE L2** | L2 개방 승인 → allowed_scope에 L2 추가, /change-gate 반영, Gate LOCKED 유지 |
| **HOLD L2** | L2 미개방 유지 → 추가 관찰 필요 시 |

### L2 승인 시 실행 순서

1. A 판정 수신
2. `ops_state.json` `allowed_scope` 갱신 (A 편집 승인에 의거)
3. `/change-gate` L2 허용 반영
4. 회귀 재확인
5. L2 개방 증빙 기록

## 6. 더 좋은 아이디어

### L2 개방 + "L3 준비 체크리스트" 병행

L2를 열면서 동시에 "L3 승인 요청 시 필요한 체크리스트"를 고정해두면, 이후 L3 판단이 빨라집니다.

L3 준비 체크리스트 예시:

- 전체 회귀 PASS

- Flower 금지 task 미발송 확인
- 변경 목적/범위/영향 문서화 (재진입 검토 템플릿)
- A 승인
