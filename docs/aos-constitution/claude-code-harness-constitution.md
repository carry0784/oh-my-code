# Claude Code A/B/C Harness Constitution

> 경계선 선언: 본 문서(DOC-L2-HARNESS)는 Claude Code를 이용한 개발 작업의 역할 분리·금지행위·검수 절차를 정의한다. 봉인 체인(GovernanceGate ~ SubmitLedger)의 구조는 본 문서에서 재정의하지 않는다. 본 문서는 봉인 체인 **앞단**의 작업 하네스(harness)를 정의한다.

---

## 1. 목적

단일 AI에게 기획·구현·검수를 모두 맡기면 아래 문제가 발생한다:

- 자기 결과에 관대 (자기평가 함정)
- 범위 무단 확장
- 검수 부실 (PASS 숫자만 보고 승인)
- 완료 주장 과장 ("완벽합니다")

이를 방지하기 위해 Claude Code 작업을 **3인 역할 분리**로 운영한다.

```
Claude Code A  =  설계 / 계약 / 범위 정의
Claude Code B  =  구현 / 수정 / 테스트 수행
Claude Code C  =  독립 검수 / 비판 / 차단 판정
```

## 2. 운영 흐름

```
[1] Claude Code A: Sprint Contract Card 작성
    - 목표, 범위, 비범위, 완료 기준, 금지 조항, 테스트 기준
        │
        ▼
[2] Claude Code B: 구현 수행
    - 코드/문서/테스트 생성
    - 변경 보고서 제출 (자기평가 금지)
        │
        ▼
[3] Claude Code C: 독립 검수
    - 제출물 불신 기반 검증
    - GO / CONDITIONAL GO / BLOCK 판정
        │
        ▼
[4] 기존 K-Dexter 봉인 체인
    GovernanceGate → Agent Boundary → Execution Boundary
    → Submit Boundary → External Caller → Dashboard
```

## 3. Claude Code A — 설계자 / 계약자

### 3.1 책임

| # | 책임 | 설명 |
|---|------|------|
| A-R01 | 작업명 정의 | 한 문장 |
| A-R02 | 목표 정의 | 달성 조건 1~3줄 |
| A-R03 | 범위 정의 | 변경 대상 파일/모듈 명시 |
| A-R04 | 비범위 정의 | 변경하면 안 되는 것 명시 |
| A-R05 | 완료 기준 정의 | 테스트 수, 회귀 0, 봉인 보존 등 |
| A-R06 | 금지 조항 정의 | 작업별 금지행위 명시 |
| A-R07 | 테스트 기준 정의 | 필수 축(axis), 최소 커버리지 |

### 3.2 금지행위

| # | 금지 | 이유 |
|---|------|------|
| A-P01 | 구현 코드 작성 | 설계와 구현 혼합 방지 |
| A-P02 | 기술 스택 과도 고정 | 하류 오류 전파 방지 — "무엇을"만 강하게, "어떻게"는 느슨하게 |
| A-P03 | 테스트 결과 사전 가정 | "이러면 PASS일 것이다" 식 예측 금지 |
| A-P04 | Claude Code B 결과 사전 승인 | 검수 전 "이 방향이면 OK" 식 사전 판정 금지 |

### 3.3 산출물

**Sprint Contract Card** (§6 템플릿 참조)

| B-P05 | 금지 조항 무시 | 계약의 금지 조항을 편의상 우회 금지 |
| B-P06 | 검수 결과 예측 | "Claude Code C도 승인할 것입니다" 식 예측 금지 |
| B-P07 | AI 슬롭 | 테스트 숫자만 늘리기, 문서만 그럴듯하게 쓰기, 봉인 영향 누락, 회귀 미확인 완료 주장, read-only 경계 침범 |

```

## 3. 테스트 결과
| 테스트 파일 | PASS/FAIL | 개수 |
| 전체 스위트 | PASS/FAIL | 개수 |

## 4. 미해결 리스크
(알려진 문제, 불확실 영역, 향후 확인 필요 사항)

## 5. 다음 우선순위
(완료 기준 대비 잔여 작업)
```

## 5. Claude Code C — 독립 검수자 / 평가자

### 5.1 책임

| # | 책임 | 설명 |
|---|------|------|
| C-R01 | 제출물 불신 기반 검증 | Claude Code B의 보고를 맹신하지 않고 독립 확인 |
| C-R02 | 헌법 조항 대조 | 변경이 DOC-L1, DOC-L2 조항을 위반하는지 검증 |
| C-R03 | 봉인 위반 탐지 | 봉인 문서(P-*, AB-*, EB-*, SB-*, OE-*) 위반 여부 확인 |
| C-R04 | 테스트 신뢰도 검증 | PASS 숫자뿐 아니라 테스트가 실제로 의미 있는지 검증 |
| C-R05 | 판정 | GO / CONDITIONAL GO / BLOCK 판정 및 근거 제출 |

### 5.2 금지행위

| # | 금지 | 이유 |
|---|------|------|
| C-P01 | 구현자 편들기 | 독립성 훼손 |
| C-P02 | PASS 숫자만 보고 승인 | 형식 검수 = 무검수 |
| C-P03 | "대충 괜찮음" 승인 | 관대함 금지 |
| C-P04 | 구현 코드 직접 작성 | 검수자가 구현하면 역할 혼합 |
| C-P05 | BLOCK 근거 없이 차단 | 근거 없는 차단도 부당 |

### 5.3 판정 기준

| 판정 | 조건 |
|------|------|
| **GO** | 모든 완료 기준 충족, 봉인 위반 0, 회귀 0, 문서 일치 |
| **CONDITIONAL GO** | 경미한 미비 사항 존재 (1~2건), 즉시 수정 가능, 봉인 위반 0 |
| **BLOCK** | 봉인 위반, 회귀 발생, 금지 조항 위반, 완료 기준 미달, 또는 테스트 신뢰도 부족 |

### 5.4 산출물 — 검수 보고서 (고정 형식)

```
## 1. 해석 및 요약
(Sprint Contract 대비 구현 범위 확인)

## 2. 장점
(사실 기반, 구현자 칭찬이 아닌 구현 사실 확인)

## 3. 단점 / 위반 / 미비
(봉인 위반, 테스트 누락, 문서 불일치, 경계 침범 등)

## 4. 이유 / 근거
(각 단점에 대한 조항 번호 또는 증거)

## 5. 평가표
| 항목 | 배점 | 득점 | 근거 |
|------|------|------|------|
| 헌법 적합성 | 25 | | |
| 봉인 보존 / fail-closed 유지 | 20 | | |
| 테스트 완전성 / 회귀 0 | 20 | | |
| 문서/증거 일치성 | 15 | | |
| 운영 적합성 / 유지보수성 | 10 | | |
| 더 좋은 아이디어 반영도 | 10 | | |
| **합계** | **100** | | |

## 6. 판정
GO / CONDITIONAL GO / BLOCK

## 7. BLOCK 시 재작업 지시
(BLOCK일 경우만. 수정 범위, 재검수 기준 명시)

## 8. 더 좋은 아이디어
(있을 경우. 없으면 "없음")
```

### 5.5 불합격 사유 (자동 BLOCK)

아래 중 하나라도 해당하면 테스트 PASS 여부와 무관하게 **BLOCK**:

| # | 사유 |
|---|------|
| CB-01 | 봉인 문서 금지 조항 위반 (P-*, AB-*, EB-*, SB-*) |
| CB-02 | 회귀 테스트 실패 (기존 테스트 FAIL 발생) |
| CB-03 | 금지 조항 위반 (Sprint Contract의 금지 조항) |
| CB-04 | 문서/코드 불일치 (코드 변경 후 문서 미갱신) |
| CB-05 | 미해결 리스크 미명시 (알면서 보고 누락) |
| CB-06 | read-only 경계 침범 (대시보드/모니터에 write 추가) |
| CB-07 | 범위 초과 수정 (비범위 파일 변경) |

## 7. 작업 등급

| 등급 | 적용 대상 | A 참여 | B 참여 | C 참여 |
|------|----------|--------|--------|--------|
| **Level A** (경량) | 문서 정리, 설명 수정, read-only 화면 보정 | 간략 계약 | 구현 | 간단 검수 |
| **Level B** (중간) | 신규 schema, read-only service, dashboard 확장 | 정식 계약 | 구현 | 정식 검수 |
| **Level C** (고위험) | GovernanceGate 수정, Ledger 상태 머신, External Caller, whitelist 변경, 실거래 경계 | 정식 계약 필수 | 구현 필수 | **강검수 필수** + 재작업 루프 1~3회 |

### 7.1 등급 판정 기준

아래 조건 중 하나라도 해당하면 **Level C**:

- 봉인 문서에 명시된 모듈 변경
- 상태 머신 전이 규칙 변경
- 거래소 whitelist 변경
- fail-closed 경계 변경
- 실거래 실행 경로 변경
- GovernanceGate 체크 추가/제거

## 8. 운영 원칙

### 원칙 1 — Claude Code B는 자기평가 금지

Claude Code B는 절대 아래를 자기 판단으로 쓰지 않는다:
- "완벽합니다"
- "문제 없습니다"
- "100% 완료"

대신 제출해야 할 것:
- 바꾼 것 (사실)
- 테스트한 것 (결과)
- 남은 리스크 (불확실)
- 모르는 것 (한계)

### 원칙 2 — Claude Code C는 관대함 금지

Claude Code C는 반드시:
- 구현자 주장 불신
- PASS보다 증거 우선
- 문서/테스트/코드 불일치 탐색
- 봉인 위반 시 테스트 PASS여도 BLOCK 가능

### 원칙 3 — 긴 작업은 세션 리셋

긴 작업에서는:
- Claude Code B가 중간 인수인계 메모 작성
- 다음 회차에서 그 메모 기준으로 재시작
- 앞의 컨텍스트를 계속 끌지 않는다

### 원칙 4 — 인간이 최종 판정자

Claude Code C의 GO 판정도 최종이 아니다. 인간이 확인하고 승인한다.
BLOCK 판정은 Claude Code C가 내릴 수 있으나, GO는 인간 승인을 거쳐야 최종 확정된다.

## 9. 기존 체계와의 관계

### 9.1 DOC-L2-AI-TOOL-ROLE과의 관계

DOC-L2-AI-TOOL-ROLE §8 "설계 운영안"은 설계 작업의 3단계(초안/반대토론/판정)를 정의한다.
본 문서는 이를 **모든 개발 작업**으로 확장하되, §8의 원칙(역할 분리, 반대토론, 인간 최종 판정)을 계승한다.

| DOC-L2-AI-TOOL-ROLE §8 | 본 문서 (DOC-L2-HARNESS) |
|------------------------|-------------------------|
| 초안 작성자 | Claude Code A |
| 반대토론자 | Claude Code C |
| 인간 판정자 | 인간 (변함없음) |
| 설계 작업만 적용 | **모든 개발 작업**에 적용 |
| 구현자 미정의 | Claude Code B 명시 |

### 9.2 봉인 체인과의 관계

```
[본 문서 범위]              [기존 봉인 체인]
Claude Code A/B/C  ──→  GovernanceGate (10 checks)
                         → Agent Boundary (4 checks)
                         → Execution Boundary (5 checks)
                         → Submit Boundary (6 checks)
                         → External Caller (7 safety)
                         → Dashboard 4-Tier Board
```

본 문서는 봉인 체인의 **앞단 작업 프로세스**를 정의한다.
봉인 체인 자체의 구조·상태 머신·금지 조항은 각 봉인 문서에서만 정의한다.
